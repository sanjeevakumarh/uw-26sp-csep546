def vanilla_solution(x: Vector, y: Vector, A: Matrix, B: Matrix) -> Vector:
    # Transpose A and B
    A_T = vanilla_transpose(A)
    B_T = vanilla_transpose(B)

    # Add A + A^T (element-by-element)
    A_plus_AT = []
    for i in range(len(A)):
        row = []
        for j in range(len(A[0])):
            row.append(A[i][j] + A_T[i][j])
        A_plus_AT.append(row)

    # Compute (A + A^T)x
    term1 = vanilla_matmul(A_plus_AT, x)

    # Compute B^T y
    term2 = vanilla_matmul(B_T, y)

    # Add the two vectors
    result = []
    for t1, t2 in zip(term1, term2):
        result.append(t1 + t2)

    return result

def numpy_solution(
    x: np.ndarray, y: np.ndarray, A: np.ndarray, B: np.ndarray
) -> np.ndarray:
    return (A + A.T) @ x + B.T @ y

Part C - 
Both implementations perform the same $O(n^2)$ work (matrix-vector multiplies, matrix addition). The algorithmic complexity is identical however vanilla Python's per-element interpreter overhead dominates at scale, while NumPy's compiled routines execute the same $O(n^2)$ work in optimized native code, in other words every multiplication or addition ressults in one call from python to C code, whereas NumPy matrix operation is once per matrix operation. Underlying data structures used appear to be different as well, as the vanilla versio uses Python lists, and the float objects are not gauranteed to be in contiguous blocks of memory, however NumPy arrays are stored contigously resulting in better cache utilization.
